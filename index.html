OOP — The Four Pillars

Object-oriented design as I apply it in Django/FastAPI codebases.

Encapsulation

Bundle data with the methods that operate on it, and hide internal state behind a controlled interface — invalid states become unrepresentable.

class Account:
    def __init__(self):
        self.__balance = 0   # hidden

    def deposit(self, amt):
        if amt > 0:
            self.__balance += amt

    @property
    def balance(self):
        return self.__balance

Abstraction

Expose what a component does, hide how. Callers depend on an interface, not on Stripe/Razorpay internals.

from abc import ABC, abstractmethod

class PaymentGateway(ABC):
    @abstractmethod
    def pay(self, amount): ...

# swap providers without
# touching business logic

Inheritance

An is-a relationship for sharing behavior. Keep hierarchies shallow; prefer composition when the relationship isn't truly is-a.

class BaseNotifier:
    def send(self, msg): ...

class EmailNotifier(BaseNotifier): ...
class SmsNotifier(BaseNotifier): ...

Polymorphism

One interface, many behaviors — via method overriding or Python duck typing. The caller never branches on type.

def notify_all(notifiers, msg):
    for n in notifiers:
        n.send(msg)
# same call, different behavior
# per concrete class
Low-Level Design — Class Relationships

The 5 relationships every UML diagram is built from, ordered strongest → weakest coupling.

Inheritance is-a

Car is a Vehicle. Subclass inherits contract + behavior. Strongest coupling — changes in the base ripple down.

Composition owns-a

House owns Rooms. Part cannot outlive the whole — delete the house, rooms go with it. UML: filled diamond.

Aggregation has-a

Team has Players. Whole-part, but parts live independently — a player survives the team disbanding. UML: hollow diamond.

Association knows-a

Doctor knows Patient. Two independent objects hold references to each other; no ownership either way.

Dependency uses-a

OrderService uses PaymentGateway as a method parameter. Weakest, method-level coupling — ideal for injection.

Design Patterns I Reach For
Strategy — swap algorithms at runtime

Define a family of interchangeable algorithms behind one interface. This is the backbone of the Business Rule Engine I built at Jio Finance: each eligibility check is a strategy, so business teams change rules without code deployments.

Observer — publish/subscribe events

Subjects notify subscribers on state change without knowing who they are. At system scale this is Kafka: loan-status events published once, consumed independently by KYC, notifications, and audit services.

Factory — centralize object creation

Callers ask for "a loan product of type X" and get the right concrete class without knowing it exists. Keeps if/elif type-switching out of business logic and makes adding new types a one-file change.

Singleton — exactly one instance

One shared instance with global access — DB connection pools, config objects. In Python, a module-level instance is usually the cleanest singleton; use it sparingly since it hides dependencies.

Decorator — layer behavior on without modifying code

Wrap an object/function to add responsibilities dynamically. Python's @decorator syntax makes this native — how I implement rate limiting, auth checks, and retry logic as reusable middleware.

Builder — step-by-step construction

Construct complex objects piece by piece instead of a 12-argument constructor. Useful for assembling loan applications where fields arrive across multiple workflow steps.

Classic LLD Problems
Parking Lot — composition + strategy
  • Entities: ParkingLot → Floors → Spots (composition); Vehicle → Car/Bike/Truck (inheritance).
  • Spot allocation: a SpotAllocationStrategy (nearest-first, compact-first) — Strategy pattern.
  • Fees: FeeCalculator per vehicle type + duration; Ticket ties vehicle, spot, and entry time together.
Rate Limiter — token bucket vs sliding window
  • Token bucket: refill N tokens/sec, allow bursts up to bucket size — my default choice.
  • Sliding window log/counter: smoother limits, more memory.
  • Distributed: counters in Redis with atomic Lua scripts — same approach I used for API rate limiting in production.
Splitwise — strategy over split types
  • Entities: User, Group, Expense, Split; Split → EqualSplit / ExactSplit / PercentSplit (Strategy).
  • Balance sheet: pairwise net balances per user; validation per split type.
  • Simplify debts: greedy max-debtor → max-creditor matching to minimize transactions.
Notification System — observer + factory
  • Channel factory: Email/SMS/Push notifiers behind one interface.
  • Observer: domain events trigger notifications without coupling business logic to channels.
  • Reliability: queue + retry with exponential backoff and dead-letter handling — mirrors my Kafka pipelines.
Networking

What actually happens between a client and my APIs.

OSI & TCP/IP Models

7 conceptual layers (OSI) vs the practical 4-layer TCP/IP stack: Application (HTTP/DNS) → Transport (TCP/UDP) → Internet (IP) → Link. Debugging = knowing which layer is lying.

TCP vs UDP

TCP: connection-oriented, ordered, reliable (retransmission, congestion control) — APIs, databases. UDP: fire-and-forget, low latency — DNS, streaming, gaming.

3-Way Handshake & TLS

SYN → SYN-ACK → ACK establishes a TCP connection; TLS handshake on top negotiates keys for HTTPS. Connection reuse (keep-alive, pooling) is why my services don't pay this cost per request.

HTTP & REST

Stateless request/response: methods (GET/POST/PUT/PATCH/DELETE), status codes (2xx/4xx/5xx), idempotency. REST maps resources to URLs — the contract behind my 25+ production APIs.

DNS Resolution

Browser cache → OS cache → recursive resolver → root → TLD → authoritative nameserver. TTLs control propagation — why DNS changes "take time".

WebSockets & Load Balancing

WebSockets upgrade HTTP to a persistent full-duplex channel for real-time updates. Load balancers route at L4 (TCP) or L7 (HTTP-aware) — Nginx does L7 in my deployments.

Operating Systems

Processes, threads, and scheduling — the layer under every backend I ship.

Processes vs Threads

A process owns its address space; threads share one within a process — cheaper to create/switch, but shared state needs synchronization. Python's GIL is why I use multiprocessing (or Celery workers) for CPU-bound work and threads/async for I/O.

CPU Scheduling

FCFS (simple, convoy effect), SJF/SRTF (optimal avg wait, needs burst prediction), Round Robin (fair time slices — quantum size matters), Priority (starvation → aging), MLFQ (what real OSes approximate). Preemptive vs non-preemptive is the key axis.

Synchronization & Deadlock

Race conditions → mutex (one owner) vs semaphore (N permits). Deadlock needs all 4: mutual exclusion, hold-and-wait, no preemption, circular wait — break any one (e.g., lock ordering) to prevent it.

Context Switching & IPC

Switching saves/restores registers, stack, and page tables — pure overhead, so fewer, busier workers often beat many idle ones. IPC: pipes, shared memory, message queues — Kafka/Redis are the same ideas at distributed scale.
